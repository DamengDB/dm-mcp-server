前端使用指南元数据管理

元数据管理

元数据管理用于浏览数据库对象结构，控制 AI 可见的数据范围，并为数据库对象补充业务注释，帮助 AI 更准确地理解和使用数据库。

页面布局

元数据管理页采用三栏布局：

  • 左侧对象树 — 层级展示数据源 → Schema → 表/视图 → 列
  • 右侧对象表格 — 展示当前选中层级下的对象列表
  • 底部批量操作栏 — 选中多个对象后弹出，支持批量设置策略和注释

浏览对象树

点击左侧树形结构的节点，逐级展开数据库对象：

数据源名称
├── 模式
│   ├── 表
│   └── 视图
└── ...

点击某个节点后，右侧表格展示该层级下的所有对象。通过面包屑导航可以快速返回上级层级。

访问策略

访问策略决定 AI 在调用 MCP 工具时能否”看到”某个数据库对象。每个对象（Schema、表、视图、列）都可以单独设置策略。

三种策略的含义

策略效果
允许（allow）AI 可以访问该对象，MCP 元数据查询会将其包含在返回结果中
拒绝（deny）AI 无法访问该对象，MCP 元数据查询会将其从返回结果中排除
继承（inherit）默认状态，继承上级对象的策略。Schema 继承自数据源，表/视图继承自 Schema，列继承自表

设置策略

在右侧表格的”访问策略”列中，点击下拉选择框即可为单个对象设置策略。

策略设置会实时生效：下一次 AI 客户端请求元数据时，MCP 服务会返回最新的可见对象集合。

批量设置策略

在右侧表格中勾选多个对象后，底部弹出批量操作栏。点击”批量允许”或”批量拒绝”可以一次性为选中的对象设置相同策略；点击”批量清空”则将其恢复为继承状态。

业务注释

业务注释是你为数据库对象添加的可读描述，AI 客户端在获取元数据时会同时收到这些注释，从而更准确地理解表和列的业务含义。

编辑注释

在右侧表格的”注释”列中，点击编辑按钮打开注释对话框，输入描述后保存。

注释与 CLI 帮助的约定

编辑注释时建议遵守下面这一约定，它与本地 dmctl 展示帮助的方式一致：

  • 第一行（从开头到第一个换行符为止）会作为该条目在 CLI 工具列表（例如执行 dmctl --help 时列出的各命令一行摘要）中的简短说明，适合写一句能概括用途的话，方便在终端里快速扫读、区分不同命令。
  • 整段注释（第一行加上后续所有行）会出现在针对该命令的 --help 输出里，作为完整说明；第二行起可写参数含义、使用注意、典型场景等细节，避免把过长内容挤在第一行。

因此：首行尽量短而准，详细内容从第二行开始写，列表里清爽，--help 里信息又够用。

批量设置注释

勾选多个对象后，使用底部批量操作栏的”批量注释”功能，可以为选中的对象统一添加相同的注释内容。

注释生效方式

业务注释通过 MCP 工具的元数据查询接口返回给 AI 客户端。当 AI 询问”数据库里有哪些表”或”这个字段是什么意思”时，MCP 服务会将你添加的注释作为对象描述的一部分返回。

使用建议

从粗到细控制可见范围

建议先在大粒度（Schema 级别）设置策略，再针对特殊对象在细粒度（表或列级别）进行调整。例如：

  1. 将业务 Schema 设为”允许”，将系统/测试 Schema 设为”拒绝”
  2. 对于允许访问的 Schema 中的敏感表（如用户密码表），单独设为”拒绝”
  3. 对于允许访问的表中的敏感列（如身份证号列），单独设为”拒绝”

为高频使用的对象添加注释

优先为核心业务表和容易混淆的字段添加注释，例如：

  • 表名缩写含义不明确的表（如 T_CUS_ORD → “客户订单表”）
  • 业务含义与字段名不一致的列（如 status 字段的实际取值为 0/1/2，分别代表”待处理/处理中/已完成”）
  • 存在多个相似表时，说明各自的使用场景和区别

定期审查

随着数据库结构的演进，建议定期审查元数据管理中的策略和注释，确保其与当前业务保持一致。
