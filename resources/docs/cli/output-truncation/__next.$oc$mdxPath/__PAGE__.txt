# 输出截断

工具返回的 JSON 在写到标准输出前会按规则截断。阈值和落盘目录来自各工具 `setup` 写入的缓存：dmctl 见 [dmctl 配置](./dmctl-config)，dmctlx 见 [dmctlx 配置](./dmctlx-config)。缓存里对应字段名为截断限制、截断文件输出目录。

截断限制为 `0` 时，不做截断，结果原样输出。

分两类：过长的字符串；同时含 `columns` 与 `records` 键的对象（按表处理）。落盘成功时，标准错误对每个文件打印一行 `[INFO] 完整内容已保存: …`。

目录不存在时程序会尝试创建；创建或写入失败时，标准错误可能出现 `[WARN]`，标准输出里可能出现「文件保存失败」一类提示。

## 过长的字符串

任意深度的 JSON 字符串，若字符个数大于截断限制：

- 标准输出：保留前面一段，接上 `... [完整内容: 绝对路径]`；写文件失败则为 `... [完整内容: 文件保存失败]`。
- 终端里展示的大约前 1024 个字符，并尽量在换行处截断。展示长度与「截断限制」不是同一个量：限制用来决定是否落盘及截断策略，终端展示仍按约 1024 字处理。
- 落盘文件名形如 `dm-truncated-{工具名}-{时间戳}-{短随机}-{序号}.{扩展名}`，一次调用多文件时序号递增。内容为合法 JSON 时扩展名为 `.json`（格式化写入），否则为 `.txt`。

## columns + records

对象中同时存在名为 `columns` 与 `records` 的数组时，按表处理（键名须完全一致）。`records` 每行一般为与 `columns` 顺序一致的值数组，例如：

```json
{
  "columns": ["id", "name", "dept_id"],
  "records": [
    [1, "Alice", 10],
    [2, "Bob", 20]
  ]
}
```

### 表何时截断

先把表排成内部约定的一整段紧凑文本（`columns` 一行，每条 `record` 一行），再比较长度与截断限制。超过限制才裁行并落盘。

### 与长字符串截断的区别

- `records`：删掉后面的行，只保留前面若干行；保留行数与表头、前几行长度有关，并受约 1024 字展示预算约束（至少一行）。单元格内超长字符串先按上一节规则处理。
- 表对象会多一个 `_truncated` 字符串字段：成功落盘为 `完整内容已保存: 绝对路径`，失败为 `完整内容: 文件保存失败`。不会在单元格后接 `... [完整内容: …]`，那种形式只用于普通字符串字段。
- 落盘文件内为整张表的紧凑文本，扩展名 `.json`。

若只有一行且该行很长，标准输出里该行仍可能很长，以落盘文件内容为准。

## 示例

长字符串（标准输出节选）：

```json
{
  "success": true,
  "data": "id|name|…\n1|Alice|…\n… [完整内容: C:/work/truncated/dm-truncated-exec_query-1716123456-a1b2c3-0.txt]"
}
```

标准错误示例：

```text
[INFO] 完整内容已保存: C:/work/truncated/dm-truncated-exec_query-1716123456-a1b2c3-0.txt
```

表裁行（字段顺序以实际输出为准）：

```json
{
  "columns": ["id", "name"],
  "records": [[1, "Alice"]],
  "_truncated": "完整内容已保存: C:/work/truncated/dm-truncated-some_tool-1716123456-b3c4d5-1.json"
}
```
