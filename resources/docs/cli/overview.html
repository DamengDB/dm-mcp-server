CLI 工具工具总览

CLI 工具总览

命令行侧提供两个核心程序，以满足不同场景下的数据库控制需求：

  • dmctl：面向远端的轻量客户端，与已部署的 MCP 服务配合工作，具体能力与权限边界主要由服务端提供。
  • dmctlx：自包含的完整形态，把常用能力收在同一分发单元里，在本机即可完成对数据库的访问与调用，不依赖外部的 MCP 服务。

关于命名中的 ctl

文件名里的 ctl 通常作两层理解：

  1. Control：强调是面向数据库操作的命令行控制入口，而不是泛称的「客户端」或「代理」。
  2. Context Tool Layer：在上层应用与数据库引擎之间，把元数据、诊断、SQL 等封装成可调用的工具。 (注：dmctlx 中的 x 表示本地/便携扩展变体)

核心对比与选型

特性dmctldmctlx
定位远端 Context Tool Layer本地便携 Context Tool Layer
角色轻量客户端，对接 MCP 服务自包含，本机独立完成访问
外部依赖需要可用的 MCP Server不需要 MCP Server
认证方式Token / API Key无（凭据在本地配置里）

如何选用？

  • 选择 dmctl：已有 MCP 服务，需要在网关侧做集中鉴权、统一出口，或需要由平台统一维护数据库的业务注释与 AI 可见性。
  • 选择 dmctlx：没有 MCP 服务，仅需在本机直接对数据库执行命令进行快速访问与调试。

能力与形态

为什么仍要用终端 CLI？

无论是远端经由 MCP 还是本机自包含路径，终端 CLI（多级子命令 + JSON 标准输出）在与 AI 结合或自动化场景中都具备显著优势：

  • 节省 AI 上下文窗口：将包含海量工具和参数模式的「整本说明书」长期放入对话上下文会严重消耗 Token 并增加幻觉概率。在终端按需查看 --help，每次只传递当前调用的参数，可大幅提升对话效率。
  • 避免缓存过期：服务端增删工具后，若 AI 对话仍依赖旧缓存会导致调用失败。CLI 实时拉取能保证能力列表的准确性。
  • 易于脚本集成：标准输出格式为 JSON，非常便于管道（Pipeline）与外部脚本进行自动化解析处理。

工作模式差异

1. dmctl（远端模式）

dmctl 将 MCP 服务端注册的工具映射为多级子命令（命令分组类似 aws 或 kubectl）。

  • 机制：配置向导从服务端拉取工具元数据并写入本地缓存；日常调用只带本次子命令及参数。
  • 优势：可充分利用 MCP 控制台侧的元数据管理能力。平台可统一为表或列补充业务注释、按策略控制 AI 可见的对象范围，大幅降低 AI 理解数据库上下文的门槛。
  • 特性：支持对超出阈值的长字段进行截断并落盘（详见 输出截断）。

2. dmctlx（本地模式）

dmctlx 不连接 MCP Server，子命令与参数随本程序版本原生提供。

  • 机制：首次运行通过 dmctlx setup 将配置写入可执行文件同目录下的 dmctlx-cache.bin。
  • 局限：由于不经过服务端，工具描述主要依赖数据库系统目录自带的注释。若库内长期缺乏业务注释，在进行数据分析或让模型理解对象含义时，往往需要人工查阅系统表补充上下文，前期准备成本相对更高。
